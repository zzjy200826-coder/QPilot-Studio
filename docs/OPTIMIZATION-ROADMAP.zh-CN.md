# QPilot Studio 优化路线图

## 当前判断

QPilot Studio 已经不是一个单纯的浏览器自动化 MVP，而是开始具备测试平台雏形：

- 功能测试链路已经比较完整，覆盖运行、诊断、证据、回放、对比、模板修复
- 平台侧已经有压测、环境、门禁、审批、豁免等控制面能力
- 后端构建、类型检查、测试链路当前可正常通过

当前最需要补的，不是“再加几个页面”，而是把可用性、发布闭环和执行规模做扎实。

## 优先级排序

### P0：立即处理

- 修复中文资源和文档漂移，保证产品可信度
- 把发布对象建模补齐，建立 `scenarioId / releaseId / buildId / commitSha` 之间的稳定关联
- 为平台资源补齐编辑、删除、审计和版本化能力
- 给 web 端补上核心页面和核心文案的自动化测试

### P1：形成完整平台闭环

- 从单活运行升级到排队和多 worker 并发
- 增加统一的 Release Detail / Evidence Hub 视图
- 将 Gate Center 的判定从标题匹配升级为基于稳定对象的真实归因
- 把压测执行面从本地分片升级到真实远端 injector

### P2：面向团队协作扩展

- 对接 CI/CD、Webhook、ChatOps、Issue Tracker
- 引入身份、角色、审批边界和审计留痕
- 拆分超大文件，沉淀更稳定的前后端模块边界

## 分阶段策略

### 第一阶段：修信任感

目标：让用户感受到“这个平台能稳定使用”。

- 修复中文乱码、统一术语、更新 README 和产品叙述
- 为运行详情、报告、门禁等关键页面增加前端测试
- 收敛重复接口和过渡态命名，降低认知成本

完成标志：

- 中文模式不再出现乱码或中英混杂的关键入口文案
- README 与当前实际页面和能力一致
- web 端至少覆盖关键文案与核心格式化逻辑的自动化测试

### 第二阶段：修发布闭环

目标：让平台可以回答“这个版本能不能发”。

- 引入发布对象和构建对象的统一关联
- 让功能测试、压测、门禁、审批围绕同一个 release 聚合
- 补齐 Release Detail 页面，统一展示证据、风险、豁免、审批轨迹

完成标志：

- 每个 release 都能追溯到对应的功能运行、压测结果和门禁结论
- Gate verdict 不再依赖脆弱的标题或自由文本匹配

### 第三阶段：修平台规模

目标：让平台具备多人协作和持续集成接入能力。

- 为功能运行增加队列与并发执行能力
- 为压测执行增加远端 worker 与 injector 池
- 引入 CI 回调、Webhook 触发和审批自动化

完成标志：

- 可同时处理多个运行任务，不再受单活模型限制
- 压测结果能区分控制面与执行面，并能追踪 worker 级健康度

## 建议的近期执行顺序

1. 中文体验修复
2. README 与蓝图文档更新
3. Web 端关键测试补齐
4. 发布对象建模
5. Release Detail / Evidence Hub
6. 多运行并发与真实分布式压测
